## What it does

EPM Wizard's "coder" role generates and edits Oracle-EPM artifacts. This model
is trained on two paired tasks:

1. **Build** ‚Äî a natural-language request (e.g. *"a monthly workforce form for
   the US entity"*) ‚Üí a complete, schema-valid `FormSpecification` JSON.
2. **Edit** ‚Äî an existing spec plus an edit instruction ‚Üí the updated spec JSON.

Every training completion was produced against real `TenantMetadata`
(the `DemoConnector` `MCWPCF` fixtures) and **kept only if `validate_form`
reported no blocking errors**, so the labels are guaranteed schema-valid.
Phrasings are honestly tagged *supported* vs *paraphrase* by running the
deterministic parser; edit labels come from actually running
`form_nlu.apply_edit`.

> **Base-model note:** the plan targeted `Qwen2.5-Coder-32B-Instruct`, but
> Together does not offer the Qwen **Coder** family for fine-tuning (its
> model-limits endpoint returns 404 for all three Coder sizes). We used the
> closest fine-tunable base ‚Äî `Qwen2.5-32B-Instruct`, same 32B size and Qwen2.5
> generation, a strong general coder. Confirm the current list anytime with
> `python -m scripts.launch_finetune --list-models`.

## Intended use & limitations

**Intended use:** power EPM Wizard's coder role ‚Äî turning natural-language
requests into schema-valid `FormSpecification` JSON, and applying NL edits to
existing specs, within the app's own validation loop.

**Limitations ‚Äî read before quoting metrics:**

- **Synthetic, single-application data.** The corpus is template-derived from
  one demo application (`MCWPCF`). The sub-0.004 eval loss reflects how regular
  that data is ‚Äî the model learned the template distribution, **not** a measure
  of production quality.
- **Real-world quality is unvalidated.** No evaluation yet on real tenant
  metadata or unseen phrasings. Treat v1 as proof the *pipeline* works
  end-to-end, which it now does.
- **General base, not code-specialized.** Uses `Qwen2.5-32B-Instruct` because
  the code-specialized Coder-32B is not fine-tunable on Together.
- **Serving economics.** A 32B fine-tune is only cheap to serve if the base
  qualifies for Together **Serverless Multi-LoRA**; otherwise a dedicated
  endpoint runs ~$4,700/mo. Per `OPENCLAW_PLAN.md` ¬ß1, prefer **stock model +
  RAG grounding** until quality demonstrably requires tuning.

**Path to a v2 worth shipping:**
1. Export a *real* corpus with `scripts.export_training_data` (validated specs,
   conversations, snapshot rule bodies).
2. Evaluate v1 and stock+RAG against a held-out set of real specs.
3. Re-tune only if it beats stock+RAG *and* serverless-LoRA eligibility is
   confirmed for the base.

## Using / reproducing

**Get the fine-tuned model name** (once `completed`):

```bash
python -m scripts.launch_finetune --status ft-b40da469-ec43
```

The `outputName` is what you point inference or serving at (typically
`<your-together-username>/Qwen2.5-32B-Instruct-epm-coder-<hash>`).

**Download the weights** (`together` CLI, key set):

```bash
# small ‚Äî the LoRA adapter you trained (needs the base to run):
together fine-tuning download ft-b40da469-ec43 --checkpoint-type adapter -o ./epm-coder-adapter
# full standalone model, base + adapter merged (~60 GB):
together fine-tuning download ft-b40da469-ec43 --checkpoint-type merged -o ./epm-coder-merged
```

**Reproduce the run from scratch:**

```bash
python -m scripts.launch_finetune --build --val-split 0.1 \
    --model Qwen/Qwen2.5-32B-Instruct --launch
```

Same corpus seed ‚Üí identical training data. See `docs/FINETUNING.md` for the
full runbook.
